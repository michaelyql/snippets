# FIX

## Introduction
- FIX = **F**inancial **I**nformation e**X**change
- Created to replace phone calls between broker-dealers which were susceptible to human error like routing to the wrong trader, or losing information when the call was hung up. 
- FIX created new messages such as the IOI (indication of interest) to buy or sell a stock.
- FIX is widely used by both the buy side (institutions) as well as the sell side (brokers/dealers).

## Technical Details

### Message Structure

- Uses tag=value encoding
- **tag** indicates the meaning of the field
    - For example, tag `48` is `SecurityID`, a string that identifies the security.
    - Tag `22` is `IDSource`, an integer that indicates the identifier class being used.
- **value** is an **array of bytes**
    - The values may be in plain text or encoded as pure binary (in which case the value is preceded by a length field).
- Fields are separated by **SOH** (start of header) character (ASCII `0x01`)
-  FIX defines meanings for most tags, but leaves a range of tags reserved for private use between consenting parties (**custom** tags, usually >= 5000)
- Some fields are **mandatory** while others are **optional**
- The *ordering* of fields within the message is generally unimportant
    - However, **repeating groups** are preceded by a count, and encrypted fields are preceded by their length.
- Each FIX message has a **header**, **body**, and **trailer/tail**
- Header contains three fields: 8(`BeginString`), 9(`BodyLength`), and 35(`MsgType`)
    - From FIX5.0, header contains 5 or 6 fields, 49(`SenderCompID`), 56(`TargetCompId`) and optional 1128(`ApplVerId`)
    - The body starts at tag 35 (inclusive) until tag 10 (exclusive), including SOH delimiters
- Content of body is defined by the message type
- Trailer contains 10(`Checksum`) expressed as a three digit integer e.g. `10=002`
    - Computed by summing the ASCII values of all characters in the message (including SOH characters) up till the checksum (not inclusive), then modulo 256
- There are two main groups of messages: **admin** and **application**.
    - **Admin** messages handle the basics of a FIX **session**
    - They allow for a session to be **started** and **terminated** and for **recovery** of missed messages.
    - **Application** messages deal with the **sending and receiving of trade-related information** such as an order request or information on the current state and subsequent execution of that order.

## Others

- There is also the **FIXML** protocol (XML encoding), but it is more used for back-office and clearing applications, settlement and reporting, rather than trading.
- **FAST** protocol (FIX Adapted for Streaming) is a protocol designed to reduce bandwidth use and latency for market data dissemination. 
    - It is used mostly for sending Multicast market data via UDP connections
    - Released in 2005
- **FIX Simple Binary Encoding (SBE)** offers a low-latency encoding standard, using primitive data types that are native to computing systems. 
    - Message encoding and decoding is therefore much lower latency than character-based protocols since no translation is needed to put data into a format that computers can use
    - Performance is more deterministic because SBE messages are constrained by templates and fixed-length data elements are preferred
    - Unlike tagvalue and FIXML, an SBE message is not self-describing
    - Released in 2020 to complement FAST encoding
